# 技术文档写作规范

## 1. 总体原则

- **读者优先**：文档为读者服务，而非展示作者知识量。每个章节问自己：读者拿到这个信息后能做什么？
- **可执行性**：所有操作步骤必须可以按顺序执行，无需读者自行推断缺失步骤。
- **精确性**：版本号、端口、路径、环境变量名必须与实际一致，禁止使用"类似于"、"大概是"等模糊表述。

---

## 2. 标题规范

```
# 一级标题：文档名称（每篇文档只有一个）
## 二级标题：主要章节
### 三级标题：子章节
#### 四级标题：仅在必要时使用，不建议更深层级
```

标题使用祈使句或名词短语，禁止使用疑问句作标题。

正确：`## 配置环境变量`
错误：`## 如何配置环境变量？`

---

## 3. 代码块规范

所有代码块必须标注语言，常用标注：

| 内容类型 | 标注 |
|----------|------|
| Shell 命令 | ` ```bash ` |
| YAML 配置 | ` ```yaml ` |
| Python 代码 | ` ```python ` |
| JSON 数据 | ` ```json ` |
| 环境变量文件 | ` ```env ` |
| 纯文本输出 | ` ```text ` |
| Dockerfile | ` ```dockerfile ` |

变量占位符统一使用 `<变量名>` 格式，例如：

```bash
docker run -e API_KEY=<your-api-key> myimage:latest
```

---

## 4. 列表使用规范

- 有序步骤使用**有序列表**（1. 2. 3.）
- 并列选项或特性使用**无序列表**（-）
- 列表项超过 5 个时，考虑拆分为多个小节
- 列表项本身不得再嵌套列表超过两层

---

## 5. 表格使用规范

以下情况使用表格：

- 配置参数说明（参数名、类型、默认值、说明）
- 环境要求对比（操作系统、版本、兼容性）
- 术语对照

表格每列必须有表头，禁止使用空表头占位。

---

## 6. 禁用词汇清单

| 禁用表述 | 原因 | 替代方案 |
|----------|------|----------|
| 简单地 / 只需 | 低估读者困难 | 直接描述操作 |
| 请注意 | 空洞前缀 | 直接写注意内容 |
| 众所周知 | 假设读者知识 | 直接陈述事实 |
| 等等 / 诸如此类 | 信息不完整 | 列举完整或说明"以下为常见示例" |
| 非常 / 很 / 十分 | 副词冗余 | 删去或替换为具体描述 |
| 笔者认为 | 不适合技术文档 | 直接陈述结论 |

---

## 7. 文档结构建议顺序

```
# 项目名称

> 一句话描述项目功能

## 功能概述          ← 是什么
## 架构说明          ← 怎么工作
## 环境要求          ← 需要什么
## 快速开始          ← 最短路径跑起来
## 详细配置          ← 参数说明
## 部署指南          ← 生产环境
## 故障排查          ← 常见问题
## 更新日志          ← 版本变化
```

并非所有文档都需要全部章节，根据文档类型取用。

---

## 8. Admonition（提示框）用法

使用 Markdown blockquote 模拟：

```markdown
> **注意**：此操作不可逆，执行前请备份数据。

> **提示**：首次部署建议使用 `--dry-run` 参数验证配置。

> **警告**：生产环境中不得使用默认密钥。
```
