# 技术文档生成器

## 概述

本 Skill 帮助 用户生成规范、专业的中文技术文档。生成结果应达到可直接用于团队交付或对外发布的质量标准。

---

## 触发场景

当用户请求以下任一类型的输出时，调用本 Skill：

- "帮我写这个项目的技术文档"
- "生成部署指南 / 架构说明 / API 参考"
- "整理一份项目复盘"
- "把这段代码/配置整理成文档"

---

## 工作流程

### 第一步：信息收集

在生成文档前，必须确认以下信息（可从用户描述、代码或配置文件中提取）：

| 字段 | 说明 | 是否必须 |
|------|------|----------|
| 项目名称 | 文档标题使用 | 是 |
| 项目类型 | RAG / Agent / 模型服务 / 知识图谱 / 其他 | 是 |
| 技术栈 | 主要框架、语言、依赖 | 是 |
| 目标读者 | 开发者 / 运维 / 业务方 | 是 |
| 文档类型 | 见下方「文档类型」 | 是 |
| 部署环境 | Docker / K8s / 裸机 / 云服务商 | 按需 |

如信息不完整，**先用合理假设生成草稿，在文档顶部标注 `[待确认]` 字段**，而非反复追问。

### 第二步：选择文档类型

根据用户需求选择对应模板（位于 `templates/` 目录）：

- `架构说明文档` → 使用 `templates/架构文档模板.md`
- `本地部署指南` → 使用 `templates/部署指南模板.md`
- `项目复盘报告` → 使用 `templates/复盘报告模板.md`
- `API 参考文档` → 内联生成，无独立模板

### 第三步：执行脚本（如适用）

如用户提供了项目目录或代码文件：

```bash
# 提取项目结构摘要
python3 scripts/extract_structure.py <项目路径>

# 验证生成文档的结构完整性
python3 scripts/validate_doc.py <输出文档路径>
```

### 第四步：生成文档

严格遵守 `reference/文档规范.md` 中的写作标准，包括：
- 标题层级、代码块语言标注
- 术语使用（参考 `reference/术语表.md`）
- 禁用表述清单

### 第五步：输出与预览

生成的文档保存为 `.md` 文件，同时可使用 `assets/doc_preview.html` 提供浏览器预览。

---

## 质量检查清单

在输出前，自检以下各项：

- [ ] 所有代码块均标注语言（```yaml、```bash 等）
- [ ] 不存在"请注意"、"需要指出"等空洞过渡语
- [ ] 命令示例可实际执行，变量用 `<尖括号>` 标注
- [ ] 端口、路径、版本号与用户提供信息一致
- [ ] 中英文术语使用与 `reference/术语表.md` 保持一致
- [ ] 文档末尾包含「常见问题」或「故障排查」章节

---

## 禁止行为

- 不得生成无法运行的虚假命令示例
- 不得在文档中使用 emoji
- 不得将所有内容平铺为单级列表，必须保持层级结构
- 不得省略「环境要求」章节
- 不得使用"简单"、"只需"、"轻松"等低估复杂度的措辞
